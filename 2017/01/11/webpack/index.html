webpack 项目配置

发表于 2017-01-11   |   分类于 Javascript

webpack 是一个更出色的前端自动化构建工具、模块化工具、资源管理工具。相比来说，传统的AMD RequireJs的各种参数设置繁琐，不易学习，很难完全掌握；个人认为和较为流行的Browserify各有优缺点，不相上下；当然还有Duo这样优秀的工具。webpack的优点简单来说是一个配置文件解决所有缓存依赖转换功能。纯从webpack的角度来说，更适合于SPA应用构建。Webpack 做的, 总体是从 entry 引入文件, 通过 loaders 编译, 从 output 输出。然后其他功能由 plugins 写的其他的方法引入, 进行分包, Hashing 等等

准备工作

  1. npm install webpack -g
  2. npm init之后，生成了package.json
  3. npm install webpack –save-dev

webpack应用

个人认为webpack最大的缺点在于官方文档黑魔法太多，底层设计机制不够透明。此处通过一个常用的webpack.config.js来大致理解webpack工作机制。

配置文件

尽管Webpack可以通过命令行来指定参数，但我们通常会将所有相关参数定义在配置文件中。一般我们会定义两个配置文件，一个用于开发时，另外一个用于产品发布。生产环境下的打包文件不需要包含sourcemap等用于开发时的代码。配置文件通常放在项目根目录之下，其本身也是一个标准的CommonJS模块。

简单配置

根据项目难易情况，可添加不同webpack项目配置。一个简单的小项目配置起来当然没那么复杂。比如需要单独打包两个html页面的js，并hash添加到页面中，经过上面的研究，会不会发现没那么复杂了。

总结

webpack当然是一个很好的前端构建工具，提供了前端在配置项目时多样化的选择。但是在很多实际应用场景，并不如期待的那么好用，比如配置多入口时，需要额外处理没有glob的情况，目录结构复杂时，file-loader的path功能无法自定义构建后的目录结构，源码复杂等等。这也可能和对webpack万物require期待过高的原因有关。经过几天的学习，发现找到一个理想的前端构建模式，仍然任重道远。
