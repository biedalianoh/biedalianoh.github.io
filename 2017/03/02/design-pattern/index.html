Javascript设计模式

发表于 2017-03-02   |   分类于 Javascript

工厂模式

核心思想是把创建实例和使用实例分开。
通过复杂程度，分为简单工厂和抽象工厂。

简单工厂

通常是专门建立一个创建实例的类，包含创建和验证接口的方法。

抽象工厂

把创建实例的方法提取成子类。

适用场合

如果不想让某个子系统，与较大的对象之间形成强耦合，而是想在运行时从许多子系统中进行挑选，工厂方法是好的选择

桥接模式

在具体实现的行为之外，通过桥梁函数，实现获取私有值，减少依赖，联结多个类等功能。
最终是以实现将抽象和其实现隔离开来的目的。

减少依赖

获取私有值

连接多个类

适用场合

把抽象和实现分离开来，有助于独立管理程序的各组成部分。
大多数桥接模式非常有用，但是会提高代码的复杂程度。

组合模式

为组合对象，叶对象等复杂组合体，添加统一执行方法。
web上动态页面界面量身定制的模式。

其中.test可能为多个，也可能为单个。

适用场合

对一类对象进行统一操作，无需大量手工遍历，只需对最顶层的对象进行操作。
各对象的耦合很松散，只要它们实现了同样的接口。
但是，如果程序层次体系很大，性能可能会受到影响。
此模式比较适合动态的HTML用户界面。

门面模式

用来创建便利函数。为执行各种复杂任务提供了一个简单的接口。
常用在把经常相伴出现的常用函数组合在一起。

适用场合

当出现反复成组出现的代码，可考虑使用门面模式。
提供了简化接口，能让程序员在编程时更轻松。

适配器模式

用来在引用新的接口时，和现有接口不兼容情况。
避免大规模修改现有代码。

适用场合

适合新添加接口和现在程序出现不兼容这种场合，用来协调语法上的差异。
用一个新的接口对现有类进行包装，有助于避免大规模改写现有程序代码。

装饰者模式

实现为对象添加功能特性，而又不用大量创建子类的模式。
所要添加的功能为一个装饰者类，这个装饰类实现了上一个实例的所有现有功能，并且可以修改，替换，添加方法。想让实例添加新的功能，只需要把上一个实例传到装饰类，生成一个新的实例即可。

需要添加功能，则添加相应的装饰类，并把实例传给装饰类来获取功能。
可多次添加功能，但是如果在中间过程中添加的装饰类中，有新增的方法，即基础装饰类中没有此方法，则在后续的装饰类中，会遗失此方法。
可在基础装饰类中做一次判断。

适用场合

需要为类添加功能，但是从该类派生子类来实现功能不太现实的话，可使用装饰者模式。
另外，想为类添加功能，但是不想改变使用该对象的代码，也可使用装饰者模式。
但是此模式会增加架构的复杂程度。

享元模式

通过把对象中的数据分为内在数据，和外在数据分离，把内在数据相同的对象替换为同一个共享对象。
解决因创建大量对象，而产生的性能问题。

适用场合

对象数目巨大，并且有一部分数据能被转化为外在数据。
编程难度和代码量大大增加。

代理模式

代理是一个对象，用来控制对另一个对象的访问。
在实例化时由于尺寸较大等原因，实例化比较费劲需要很长时间才能完成时，比较有用。

适用场合

对于类或对象需要使用大量内存，并且并不需要实例化后立即访问这些数据。
当其他程序员立即访问本体，会出错，需要文档详细。
其他的保护代理和远程代理无法在Javascript中实现。

观察者模式

检测事件的进行状态，监测事件的发生。
如事件监听器就是观察模式应用的一种。

适用场合

把特定的元素的操作提取出来，变成一个可监听的事件。不用再给每个元素进行绑定。
是对系统进行抽象的有力手段。

命令模式

通过一个中间方法对象，连接接受者和调用者，降低了两者的耦合度，便于代码的修改和维护。

适用场合

适合需要把操作和调用分离时，或需要对操作进行规范化处理时再用它。
但降低了代码的可阅读性（毕竟谁都不愿意看个代码的具体实现要逐层找半天，从调用者，到接收者），增加了代码的难度。

职责链模式

有多个对象可执行方法。可根据情况选择使用哪个方法。

适用场合

可动态的选择由哪个对象来处理请求。
